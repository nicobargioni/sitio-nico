1.1 Carga de datos

Uso el dataset Diabetic Retinopathy 224x224 (Gaussian Filtered) (mismo que el del artículo), descargado con kagglehub. Está organizado en carpetas por clase, así que uso ImageFolder. Las 5 clases son: No_DR, Mild, Moderate, Severe, Proliferate_DR — fuertemente desbalanceadas (ver Parte 2.1).

# 🧠 Prompt: "descargá el dataset con kagglehub y localizá la carpeta de imágenes por clase"
import kagglehub
ruta = kagglehub.dataset_download("sovitrath/diabetic-retinopathy-224x224-gaussian-filtered")
# La carpeta con subdirectorios por clase:
img_dir = os.path.join(ruta, "gaussian_filtered_images", "gaussian_filtered_images")
if not os.path.isdir(img_dir):
    # fallback: buscar el primer dir que tenga subcarpetas de clase
    for root, dirs, files in os.walk(ruta):
        if {"No_DR","Severe"}.issubset(set(dirs)):
            img_dir = root; break
print("Directorio de imágenes:", img_dir)
print("Clases:", sorted(os.listdir(img_dir)))
# 🧠 Prompt: "definí transforms (resize 224 + normalización ImageNet) y armá el ImageFolder"
# Transforms base (para transfer learning con pesos ImageNet)
tf_eval = transforms.Compose([
    transforms.Resize((224, 224)),
    transforms.ToTensor(),
    transforms.Normalize([0.485, 0.456, 0.406], [0.229, 0.224, 0.225]),
])
# Para train sumamos un leve augmentation (ver discusión en 2.3)
tf_train = transforms.Compose([
    transforms.Resize((224, 224)),
    transforms.RandomHorizontalFlip(p=0.5),
    transforms.RandomRotation(10),
    transforms.ToTensor(),
    transforms.Normalize([0.485, 0.456, 0.406], [0.229, 0.224, 0.225]),
])

dataset_full = datasets.ImageFolder(img_dir)           # sin transform todavía
clases = dataset_full.classes
print("Clases:", clases, "| total imágenes:", len(dataset_full))
# 🧠 Prompt: "partí en train/val (80/20) y asigná los transforms correspondientes a cada subset"
n_val = int(0.2 * len(dataset_full))
n_train = len(dataset_full) - n_val
train_set, val_set = random_split(dataset_full, [n_train, n_val],
                                  generator=torch.Generator().manual_seed(42))

# Wrapper para aplicar transform distinto a cada subset
class ConTransform(torch.utils.data.Dataset):
    def __init__(self, subset, tf): self.subset, self.tf = subset, tf
    def __len__(self): return len(self.subset)
    def __getitem__(self, i):
        img, y = self.subset[i]
        return self.tf(img), y

train_ds = ConTransform(train_set, tf_train)
val_ds   = ConTransform(val_set,   tf_eval)
print("train:", len(train_ds), "| val:", len(val_ds))

<!-- 🧠 Prompt: "explicá el uso de WeightedRandomSampler para el desbalance" -->

Desbalance (se conecta con la Parte 2.1). Como hay muchas más imágenes No_DR que Severe, uso un WeightedRandomSampler: da más probabilidad de muestreo a las clases minoritarias, para que cada batch sea más equilibrado.

# 🧠 Prompt: "calculá pesos por clase y construí el WeightedRandomSampler y los DataLoader"
# etiquetas del subset de train
train_targets = [dataset_full.targets[i] for i in train_set.indices]
conteo = np.bincount(train_targets, minlength=len(clases))
print("conteo por clase (train):", dict(zip(clases, conteo)))

peso_clase = 1.0 / np.maximum(conteo, 1)
peso_muestra = [peso_clase[t] for t in train_targets]
sampler = WeightedRandomSampler(weights=peso_muestra, num_samples=len(peso_muestra), replacement=True)

BATCH = 32
train_loader = DataLoader(train_ds, batch_size=BATCH, sampler=sampler, num_workers=2)
val_loader   = DataLoader(val_ds,   batch_size=BATCH, shuffle=False, num_workers=2)

<!-- 🧠 Prompt: "introducí el modelo con transfer learning sobre MobileNetV2" -->

1.2 Modelo — Transfer Learning con MobileNetV2

Cargo MobileNetV2 preentrenada en ImageNet, congelo el extractor de features y reemplazo el clasificador final por uno de len(clases) salidas. (En la Parte 2.2 discuto cuándo descongelar capas.)

# 🧠 Prompt: "cargá MobileNetV2 preentrenada, congelá el backbone y reemplazá el clasificador"
modelo = models.mobilenet_v2(weights=models.MobileNet_V2_Weights.IMAGENET1K_V1)
for p in modelo.features.parameters():
    p.requires_grad = False                     # congelo el backbone
n_in = modelo.classifier[1].in_features
modelo.classifier[1] = nn.Linear(n_in, len(clases))   # nueva cabeza
modelo = modelo.to(DEVICE)
print("Cabeza nueva:", modelo.classifier[1])

1.3 Entrenamiento

# 🧠 Prompt: "definí el loop de entrenamiento y validación por época y entrená pocas épocas"
criterio = nn.CrossEntropyLoss()
optim = torch.optim.Adam(modelo.classifier.parameters(), lr=1e-3)
EPOCHS = 5      # reducido para demo en Colab; subir si hay tiempo/GPU

hist = {"train_loss": [], "val_acc": []}
for ep in range(EPOCHS):
    modelo.train(); run = 0.0
    for xb, yb in train_loader:
        xb, yb = xb.to(DEVICE), yb.to(DEVICE)
        optim.zero_grad()
        loss = criterio(modelo(xb), yb)
        loss.backward(); optim.step()
        run += loss.item() * xb.size(0)
    # validación
    modelo.eval(); correct = total = 0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb, yb = xb.to(DEVICE), yb.to(DEVICE)
            pred = modelo(xb).argmax(1)
            correct += (pred == yb).sum().item(); total += yb.size(0)
    hist["train_loss"].append(run/len(train_ds)); hist["val_acc"].append(correct/total)
    print(f"época {ep+1}: train_loss={hist['train_loss'][-1]:.4f}  val_acc={hist['val_acc'][-1]:.4f}")
# 🧠 Prompt: "graficá la curva de loss de entrenamiento y la accuracy de validación"
fig, ax = plt.subplots(1, 2, figsize=(11,4))
ax[0].plot(hist["train_loss"], marker="o"); ax[0].set_title("Train loss"); ax[0].set_xlabel("época")
ax[1].plot(hist["val_acc"], marker="o", color="green"); ax[1].set_title("Val accuracy"); ax[1].set_xlabel("época")
plt.tight_layout(); plt.show()

modelo.eval(); y_true, y_pred = [], []
with torch.no_grad():

2.1 Desbalanceo de clases y métricas engañosas:

El conjunto de datos original está fuertemente desbalanceado (la clase *No_RD* tiene 1805 imágenes, frente a solo 193 de la clase *Severe*). Si un modelo mal entrenado predijera siempre que ningún paciente tiene retinopatía, ¿qué precisión (*accuracy*) obtendría? ¿Por qué esta métrica es peligrosa en el contexto médico? Expliquen qué técnica de PyTorch (ej. WeightedRandomSampler o pesos en la función de pérdida) implementarían para solucionarlo y justifiquen su elección.

<!-- 🧠 Prompt: "respondé la pregunta 2.1 sobre desbalance y métricas engañosas" -->

Respuesta 2.1. Si el modelo predijera siempre "sin retinopatía" (No_DR), como esa clase tiene 1805 de ~1998 imágenes, obtendría una accuracy ≈ 90% — altísima pero inútil y peligrosa: no detectaría ningún caso grave (Severe), que es justamente lo que importa en medicina (un falso negativo puede significar un paciente sin tratar). Por eso el accuracy engaña con clases desbalanceadas; hay que mirar recall por clase (sensibilidad), F1 macro y la matriz de confusión.

Solución elegida: usé WeightedRandomSampler (implementado arriba) para equilibrar el muestreo por batch. Como alternativa equivalente se puede pasar pesos a la pérdida (nn.CrossEntropyLoss(weight=...)), penalizando más el error en clases minoritarias. Preferí el sampler porque no cambia la escala de la loss y funciona bien con ImageFolder; con datasets muy grandes, los pesos en la loss son más baratos (no re-muestrean).

2.2 Criterio arquitectónico:

El tutorial utiliza *MobileNetV2*. ¿En qué casos de uso en el mundo real es fundamental elegir una arquitectura ligera como esta en lugar de modelos más profundos como *ResNet152*? Además, expliquen en qué escenario clínico justificarían "descongelar" (*fine-tuning*) las últimas capas convolucionales de la red base.

<!-- 🧠 Prompt: "respondé la pregunta 2.2 sobre criterio arquitectónico" -->

Respuesta 2.2. Conviene una arquitectura ligera como MobileNetV2 cuando hay restricciones de cómputo/latencia/energía: despliegue en el borde (un equipo de fondo de ojo portátil, un celular en una zona rural sin GPU), inferencia en tiempo real, o cuando el dataset es chico (un modelo enorme como ResNet152 sobreajustaría y sería caro de entrenar/servir). ResNet152 tiene sentido si hay mucho dato, GPU potente y se busca exprimir el último punto de accuracy en un servidor central.

Cuándo descongelar (*fine-tuning*) las últimas capas convolucionales: cuando el dominio objetivo es muy distinto a ImageNet (las imágenes de fondo de ojo no se parecen a fotos naturales) y se cuenta con datos suficientes. Las primeras capas detectan bordes/texturas genéricas (sirven tal cual); las últimas son más específicas, así que descongelarlas con un learning rate bajo permite re-adaptarlas a patrones médicos (microaneurismas, exudados) sin destruir lo aprendido.

2.3 Sentido del aumento de datos (*data augmentation*):

Revisen las transformaciones de imágenes aplicadas en el código original. Al trabajar con fotografías reales de fondo de ojo, ¿es anatómicamente válido aplicar giros verticales u horizontales aleatorios? Entrenen brevemente el modelo con y sin transformaciones y describan el impacto observado en la generalización.

<!-- 🧠 Prompt: "respondé la pregunta 2.3 sobre validez del data augmentation en fondo de ojo" -->

Respuesta 2.3. En fotos de fondo de ojo, el flip horizontal suele ser válido (un ojo izquierdo espejado se parece a un derecho; la patología no cambia), por eso lo incluí. El flip vertical es más discutible: invierte la orientación anatómica (posición del disco óptico y la mácula), generando imágenes poco realistas — puede meter ruido en vez de ayudar. Rotaciones leves (±10°) sí son razonables porque la cámara puede variar un poco el ángulo. La clave: el augmentation debe generar variaciones plausibles del mundo real, no cualquier transformación.

# 🧠 Prompt: "entrená brevemente con y sin augmentation para comparar la generalización"
# Experimento corto: comparar val_acc con augmentation (tf_train) vs sin él (tf_eval en train)
def entrenar_rapido(tf_para_train, epochs=2):
    ds = ConTransform(train_set, tf_para_train)
    dl = DataLoader(ds, batch_size=BATCH, sampler=sampler, num_workers=2)
    m = models.mobilenet_v2(weights=models.MobileNet_V2_Weights.IMAGENET1K_V1)
    for p in m.features.parameters(): p.requires_grad = False
    m.classifier[1] = nn.Linear(m.classifier[1].in_features, len(clases)); m = m.to(DEVICE)
    opt = torch.optim.Adam(m.classifier.parameters(), lr=1e-3); crit = nn.CrossEntropyLoss()
    for _ in range(epochs):
        m.train()
        for xb, yb in dl:
            xb, yb = xb.to(DEVICE), yb.to(DEVICE)
            opt.zero_grad(); crit(m(xb), yb).backward(); opt.step()
    m.eval(); c=t=0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb=xb.to(DEVICE); c+=(m(xb).argmax(1).cpu()==yb).sum().item(); t+=yb.size(0)
    return c/t

acc_con = entrenar_rapido(tf_train)
acc_sin = entrenar_rapido(tf_eval)
print(f"val_acc CON augmentation: {acc_con:.4f}")
print(f"val_acc SIN augmentation: {acc_sin:.4f}")

<!-- 🧠 Prompt: "interpretá el resultado del experimento de augmentation" -->

Observación. Normalmente el augmentation razonable (flip horizontal + rotación leve) mejora la generalización (sube o sostiene la val_acc y reduce el sobreajuste), porque el modelo ve más variabilidad. Si se abusa con transformaciones no realistas (flip vertical fuerte, distorsiones grandes), puede empeorar. *(Con pocas épocas la diferencia puede ser chica; con más entrenamiento se nota mejor.)*

2.4 Diagnóstico de fallos y *human-in-the-loop*:

El autor menciona que el modelo falló al predecir la etapa 3 (*Severe*). Analizando los datos y el código, ¿cuál creen que es la causa técnica principal de este fallo? Como futuros profesionales, propongan un protocolo de validación humana (*human-in-the-loop*) sobre cómo debería reaccionar el sistema en producción si la red neuronal devuelve una predicción con un nivel de confianza muy bajo.

<!-- 🧠 Prompt: "respondé la pregunta 2.4 sobre diagnóstico de fallos y human-in-the-loop" -->

Respuesta 2.4. La causa técnica principal del fallo en Severe es el desbalance extremo (solo ~193 imágenes): el modelo casi no ve esa clase, así que aprende poco sus rasgos y tiende a confundirla con Moderate/Proliferate. Se suma que las diferencias entre etapas avanzadas son sutiles y que la métrica global (accuracy) no penaliza ignorar la minoría.

Protocolo *human-in-the-loop* propuesto: en producción, calcular la confianza (probabilidad softmax máxima) de cada predicción. Si la confianza es baja (p. ej. < 0.7) o si la clase predicha es grave (Severe/Proliferate), el sistema no decide solo: marca el caso como "revisión humana requerida" y lo deriva a un oftalmólogo, mostrando la imagen y las probabilidades por clase. El modelo actúa como filtro/priorizador (triage), no como diagnóstico final. Además, esos casos revisados se reincorporan al dataset (active learning) para mejorar justamente las clases difíciles.
